Personal Multi-Agent Orchestration System

Project Owner: Nurdin
 Jenis Sistem: Personal Multi-Agent Orchestration, RAG, Knowledge Management, dan WhatsApp AI Operations
 Status: Stable Development Checkpoint
 Tujuan Dokumen: Dokumentasi alur proses, teknologi, definisi, menu, dan keterkaitan antar modul untuk kebutuhan customer dan presentasi.

1. Executive Summary

Personal Multi-Agent Orchestration System adalah platform berbasis AI yang dirancang untuk mengelola beberapa agent dalam satu sistem terpadu. Sistem ini memungkinkan pengguna menjalankan task melalui dashboard maupun WhatsApp, mengatur kemampuan agent, melakukan governance terhadap request, menyimpan knowledge dan memory, melakukan semantic retrieval berbasis RAG, serta memantau seluruh aktivitas melalui pusat audit.

Platform ini dibangun sebagai sistem lokal yang modular, scalable, dan production-oriented. Setiap modul dibuat saling terhubung, mulai dari Overview Dashboard, Tasks Center, WhatsApp Operations, Agents Governance, Skills Library, Memory Vault, sampai Settings Provider Registry.

Tujuan utama sistem ini adalah menyediakan fondasi enterprise-ready untuk:

Mengelola banyak AI agent dalam satu interface.
Memproses task dari berbagai channel seperti manual dashboard dan WhatsApp.
Mengatur batas kemampuan agent agar aman dan terkontrol.
Menyimpan knowledge dan memory untuk mendukung context-aware response.
Menggunakan RAG untuk menghasilkan jawaban yang lebih relevan.
Menyediakan audit trail untuk semua task, runtime, governance, dan retrieval.
Memudahkan monitoring operasional melalui dashboard visual yang modern.
2. Tujuan Sistem

Sistem ini dibuat untuk menjawab kebutuhan integrasi AI agent yang tidak hanya mampu menjawab prompt, tetapi juga dapat bekerja dalam workflow yang terstruktur, terpantau, dan aman.

Tujuan utama:

Multi-Agent Management
 Mengelola beberapa agent dengan role, capability, dan governance masing-masing.

Task Orchestration
 Semua request dari user diproses sebagai task yang bisa diaudit.

WhatsApp AI Integration
 WhatsApp menjadi salah satu channel utama untuk interaksi dengan agent.

Agent Governance
 Setiap agent memiliki aturan capability, allowed keyword, denied keyword, fallback, dan boundary message.

Memory Vault dan RAG
 Knowledge dan memory disimpan, di-chunk, di-embed, dan dapat diambil kembali secara semantic untuk memperkaya jawaban agent.

Knowledge Source Management
 Sistem mendukung import knowledge dari text, markdown, file, atau folder, lengkap dengan audit, versioning, diff, dan rollback.

Provider Registry
 Sistem mendukung pengaturan provider AI runtime agar agent dapat diarahkan ke model yang sesuai.

Monitoring dan Audit
 Semua aktivitas agent, task, WhatsApp, governance, dan RAG dapat dipantau dari dashboard.

4. Konsep Utama Sistem
4.1 Agent

Agent adalah unit AI yang memiliki role dan kemampuan tertentu. Contoh agent:

design-agent
research-agent
code-agent

Setiap agent dapat memiliki:

Role.
Description.
Allowed keywords.
Denied keywords.
Boundary behavior.
Fallback agents.
Linked skills.
Memory scope.
4.2 Task

Task adalah record pekerjaan yang diproses oleh sistem. Setiap input user dari dashboard atau WhatsApp akan menjadi task.

Task menyimpan:

Input user.
Agent tujuan.
Source channel.
Status.
Output agent.
Runtime provider.
Governance result.
Memory metadata.
RAG metadata.
Timestamp.
4.3 Skill

Skill adalah kemampuan spesifik yang dimiliki agent. Skill dapat digunakan untuk:

Capability mapping.
Governance matching.
Skill-to-RAG linking.
Agent behavior foundation.
4.4 Memory

Memory adalah data knowledge yang disimpan agar bisa digunakan kembali oleh agent.

Memory memiliki:

Agent owner.
Scope.
Sensitivity level.
Runtime injectable flag.
RAG enabled flag.
Linked skills.
Source reference.
4.5 Chunk

Chunk adalah potongan kecil dari memory atau knowledge source. Chunk digunakan agar data dapat diindeks dan dicari secara semantic.

4.6 Embedding

Embedding adalah representasi numerik dari teks agar sistem bisa melakukan semantic search.

4.7 RAG

RAG atau Retrieval Augmented Generation adalah proses mengambil context relevan dari Memory Vault sebelum agent menghasilkan jawaban.

4.8 Governance

Governance adalah layer kontrol yang menentukan apakah sebuah request boleh diproses oleh agent tertentu atau harus ditolak/dialihkan.

4.9 Provider

Provider adalah model AI runtime yang digunakan oleh agent untuk menghasilkan jawaban.

4.10 Runtime Adapter

Runtime adapter adalah penghubung antara input channel dan agent runtime. Saat ini terdapat dua channel utama:

Manual dashboard atau Floating Assistant.
WhatsApp.

6. Alur Proses Utama
6.1 Alur Manual Dashboard / Floating Assistant
User mengirim prompt dari Floating Assistant
↓
Sistem membuat task baru
↓
Agent target dipilih
↓
Governance mengecek apakah request sesuai capability agent
↓
Jika allowed, runtime provider dipilih
↓
Memory resolver mencari memory yang relevan
↓
RAG mengambil chunk yang paling sesuai
↓
Context diinjeksi ke prompt runtime
↓
LLM menghasilkan jawaban
↓
Jawaban disimpan ke task record
↓
Dashboard menampilkan output dan metadata


Nilai bisnis:

Semua request tercatat.
Semua output dapat diaudit.
Agent tidak menjawab di luar batas capability.
Jawaban bisa lebih kontekstual karena memakai memory/RAG.
6.2 Alur WhatsApp
User mengirim pesan WhatsApp
↓
WhatsApp runtime menerima pesan
↓
Pesan dikonversi menjadi task
↓
Agent target dipilih berdasarkan mention atau routing
↓
Governance melakukan capability check
↓
Memory/RAG retrieval dijalankan jika eligible
↓
Agent menghasilkan response
↓
Response dikirim kembali ke WhatsApp
↓
Task dan metadata disimpan ke audit


Nilai bisnis:

WhatsApp dapat menjadi channel command utama.
Semua interaksi tetap masuk ke sistem audit.
Balasan agent bisa tetap memakai governance dan RAG.
Operasional WhatsApp dapat dimonitor dari dashboard.
6.3 Alur Agent Governance
User mengirim request ke agent
↓
Sistem membaca capability contract agent
↓
Request dicek terhadap allowed keywords
↓
Request dicek terhadap denied keywords
↓
Request dicek terhadap soft allowed keywords
↓
Jika sesuai, request diproses
↓
Jika tidak sesuai, agent memberi refusal atau redirect
↓
Governance metadata disimpan ke task


Nilai bisnis:

Agent tidak menjawab semua hal secara bebas.
Setiap agent memiliki batas tanggung jawab.
Risiko jawaban yang tidak relevan dapat dikurangi.
Customer bisa melihat bahwa sistem punya kontrol keamanan dan scope.
6.4 Alur Knowledge Source Import ke RAG
User import text, markdown, file, atau folder
↓
Sistem membuat knowledge_source memory
↓
Import history dibuat
↓
Memory dipecah menjadi chunks
↓
Embedding dibuat untuk chunk
↓
Chunk menjadi searchable secara semantic
↓
Runtime RAG dapat memakai chunk sebagai context


Nilai bisnis:

Knowledge customer bisa dimasukkan ke sistem.
Knowledge dapat diaudit.
Perubahan knowledge memiliki history.
Jika terjadi kesalahan, knowledge bisa dirollback.
Agent bisa menjawab berdasarkan knowledge internal.
6.5 Alur Retrieval Lab
User memasukkan query semantic
↓
Sistem mencari chunk relevan dari Memory Vault
↓
Sistem mengecek eligibility berdasarkan scope, agent, skill, dan sensitivity
↓
Chunk dengan score tertinggi ditampilkan
↓
User dapat mengevaluasi kualitas retrieval


Nilai bisnis:

Customer bisa melihat bagaimana RAG bekerja.
Developer bisa tuning retrieval quality.
Retrieval bisa diuji sebelum dipakai production.
